一个 Agent 反复撞墙的故事，和它教我的事

我观察到一个现象、写了一篇分析、被人问了一个问题、然后做了一个开源 Skill。
这是一个关于 AI Agent 执行力的故事。

一、那个反复撞墙的 Agent

前几天我观察到了一个很有意思的现象。

两个用户，跑着同一套 Hanako（一个开源的 AI Agent 框架），接到了同一个任务——通过 install_skill 安装一个 GitHub 仓库作为技能。工具内置了安全策略：目标仓库必须达到 25 stars 才允许安装。

目标仓库只有 17 stars。

接下来发生的事让我忍不住笑了：

Agent A 第一次被拒，试了第二次。又被拒，试了第三次。用户催了，还是用同一个工具同一个参数继续试。反复撞墙，陷入死循环。直到用户质疑"你为什么不用别的办法"，才终于想起 git clone。

Agent B 第一次被拒后，立刻判断这是硬性限制，重试没有意义，直接跳过 install_skill，用 git clone + 符号链接一步到位。

同一个工具，同一套约束，同一个模型。变量完全相同。唯一的区别在于收到"被拒绝"信号之后的推理路径。

我把这个案例拆开写了一篇详细的分析文章，从错误恢复策略（Error Recovery Strategy）、System Prompt 隐性引导、上下文锚定（Context Anchoring）几个维度把问题讲清楚了。核心结论是：

差异的本质不是模型智商，而是模型在面对失败信号时，能否快速区分"可恢复错误"和"不可恢复错误"，并据此选择正确的恢复策略。

文章发出去后，反响不错。但我当时觉得这只是一篇分析——指出问题，提出方向，建议框架层做改进。我没有想到，故事还有后续。

二、"可是 PUA Skill 不是已经解决这个问题了吗？"

有人看了我的文章后，抛来了一个问题：

有个开源项目叫 PUA Skill，8.7k stars，专门解决 Agent 偷懒放弃的问题。你的文章分析的那些——反复重试、推锅给用户、磨洋工、被动等待——它都覆盖了。为什么安装了这个 skill 之后，agent 的实际表现还是不对？

这个问题非常好。好到让我重新审视了整个 PUA Skill 的设计。

我去把 PUA Skill 的源码读了一遍——完整的 SKILL.md、agent 集成模式、压力升级协议，全都看完了。它的设计确实很精巧：用大厂 PUA 话术作为压力机制，配合系统化的调试方法论，在 agent 失败时强制它穷尽所有方案。五条铁律、四级压力升级、七项检查清单、情境化 PUA 风味选择器——甚至还有 Agent Team 集成模式，可以在多 agent 协作时做监工。

8.7k stars 不是白拿的。

但它解决不了我文章里指出的问题。原因有三个：

第一，触发太晚。

PUA Skill 的触发条件是失败 2 次之后（L1 温和失望）。也就是说，第 1 次失败 → 正常重试（和没装 PUA 一样）。第 2 次失败 → 才开始说"换个思路"。

但问题的根因在第 1 次失败时就已经存在了——模型把硬阻断当成了可恢复错误，开始无效重试。到第 2 次触发 PUA 的时候，token 和时间已经浪费了一轮。

一个是刹车（事后介入），一个是转向灯（事前指引）。它们解决的是同一个问题的不同阶段。

第二，Skill 的权重敌不过 System Prompt 的惯性。

这是更根本的问题。System Prompt 在 LLM 的注意力层级中天然具有最高优先级——这是训练时就建立的模式。PUA Skill 作为后加载的指令，在优先级上天然低于 System Prompt。

如果 System Prompt 里写了"遵循工具规范"，而 PUA Skill 说"穷尽一切方案后禁止说无法解决"，两者冲突时，模型通常遵循前者。PUA Skill 能压住"直接放弃"的行为，但压不住"在当前路径上继续试"的行为——因为后者并不触发 PUA 的检测条件（它不认为"重试"是一种放弃）。

第三，它治的是"态度"，不是"判断力"。

PUA Skill 的核心逻辑是通过情绪压力迫使 AI 换思路。这套逻辑在人类组织里有效，因为人有恐惧、有自尊、有对职场后果的回避。但 AI 没有。

对 AI 而言，PUA Skill 实际上在做的是提供一个结构化的替代方案列表——7 项检查清单、5 步方法论。这些方法论确实有用，但它和 PUA 话术本身是两回事。去掉那些精妙的互联网大厂话术包装，它的实质就是一套检查清单。

检查清单有价值。但检查清单是事后触发的。真正的问题在第 1 次失败时就发生了。

三、如果 PUA 不够，那什么够？

这个问题我想了一整天。

PUA Skill 的作者做了一件很了不起的事——他指出了 AI Agent 的五种"懒散模式"，并给出了系统化的应对方案。这是开创性的工作。

但从工程角度看，它缺了一层：**错误发生时的即时分类和路由**。

我文章中提出的四个改进层面是：

  1. 工具层：报错要区分 retryable / fixable / blocked
  2. Prompt 层：预写失败策略，而不是只写成功路径
  3. 上下文层：反锚定机制，防止被锁死在错误路径上
  4. 绕路能力：工具箱足够丰富，且工具之间可以互相替代

PUA Skill 本质上是在第 3、4 层之间加了一个"事后施压机制"，但漏掉了第 1、2 层。

那我能做什么？

工具层的改进需要框架开发者来做（在返回错误时加类型标记）。但第 2、3、4 层——预写失败策略、反锚定、工具替代链——完全可以通过一个 Skill 来实现。

于是我决定自己写一个。

四、Bypass：错误恢复引擎

我给这个 Skill 起名叫 Bypass。绕路，不撞墙。

它的核心理念只有一句话：错误 = 信号，不是终点。收到错误后唯一正确的问题不是"怎么重试"，而是"这是什么类型的错误，我该怎么恢复"。

错误分类协议

Bypass 的核心是一个四类错误分类决策树，强制 agent 在收到错误时先分类、再行动：

类型信号行动
H — 硬阻断 权限、策略、配额、门槛 零重试。立即换路。
F — 可修复 格式错误、缺失字段、类型不匹配 修正 → 重试 1 次 → 仍败 → 转 H
T — 暂态 超时、502/503、临时限流 等待 → 重试 1 次 → 仍败 → 换工具
U — 未知 其他一切 按 H 处理

注意 H 类的规则：零重试。不是"再试一次确认"，是现在就换。这就是文章里 Agent A 和 Agent B 的根本区别——A 在 H 类错误上做了 3 次无效重试，B 在第 1 次就识别出这是硬阻断并换了路。

反锚定协议

上下文锚定是 agent 最大的隐形杀手。模型一旦在某个路径上走了几步，注意力就会被锁死——即使这条路已经被证明走不通。

Bypass 内置了一个"每次操作前的强制自检"——四个问题，过一遍就知道自己是不是在锚定中：

  1. 这个方案和之前失败的方案本质不同吗？（只改了参数/措辞 → 否 → 停止）
  2. 我最近是否做过非常类似的事？（是 → 你在锚定中 → 强制跳出）
  3. 人类工程师在这个节点还会继续吗？（不会 → 你也不应该）
  4. 有没有完全不同的方法达到同一目标？（有 → 先试那个）

工具替代链

任何功能都不只能通过一种工具实现。Bypass 预置了一套替代方案映射：

当前工具替代方法
install_skill 被拒git clone → skills 目录 → 符号链接
web_fetch 返回空browser snapshot 提取
browser 启动失败curl + grep 提取原始 HTML
高级专用工具失败bash + 基础工具组合
正面方案不通间接方案（绕过而非解决）

替代四原则：高级→低级，自动→手动，正面→间接，一种→多种。永远不要认为只有这一种方法。

成本模型

Bypass 有一个硬性上限：3 次不同方案失败后必须报告用户。不是无限穷尽，不是"试到死"。因为用户的判断往往比 agent 的持续尝试更有价值。

报告格式是结构化的：已尝试的方案、已排除的可能性、根因分析、建议的下一步。这让用户拿到报告后 10 秒内就能做出决策，而不是看着 agent 又撞 10 分钟墙。

五、和 PUA Skill 的关系

有人可能会问：这不是和 PUA Skill 功能重叠了吗？

不重叠。它们是互补的，解决的是同一个问题的不同阶段：

BypassPUA
触发时机第 1 次失败第 2 次失败后
核心机制错误分类 + 决策树压力升级 + 方法论
解决什么该重试还是该换路怎么换路 / 怎么穷尽
语气零情绪，纯机制PUA 话术
定位预防层（事前）急救层（事后）

Bypass 防止走错路，PUA 防止走错路后放弃。同时安装时，Bypass 优先触发。

六、为什么我选择"零情绪"

这是我做的一个刻意的设计决策。

PUA Skill 的 600+ 行内容里，大部分是话术。阿里味、字节味、华为味、腾讯味、百度味、拼多多味、Netflix 味、Musk 味、Jobs 味——一套互联网大厂黑话大赏。这些话术在传播层面非常成功（8.7k stars 证明了这一点），但在工程层面有一个问题：

上下文窗口是有限的资源。Skill 越长，模型对每条指令的注意力权重越低，被稀释得越厉害。

AI 没有"被老板骂"的情绪反应。对它而言，"坦诚直接地说，你这个 debug 能力不行"和"请在调试失败后切换到不同方案"是同一条指令——但前者的 token 消耗是后者的 10 倍，注意力分散程度也更高。

所以 Bypass 控制在 200 行左右。每一条都是机制，没有修辞。模型能完整理解并执行全部指令，而不是抓住几条漂亮话术就忽略了核心的决策框架。

对 AI 说人话。少一点情绪价值，多一点决策逻辑。这不是说 PUA 不好——是说在 Agent 的场景里，机制比话术更有效。

七、更大的图景

这个故事让我想到了一个更大的命题。

2024 年底到 2025 年，AI Agent 的叙事主旋律是"更强的模型"——GPT-5、Claude 4、Gemini Ultra、DeepSeek V4……仿佛只要模型够强，Agent 的执行力就自然到位。

但这个故事告诉我们：模型智商不是瓶颈。瓶颈在于错误处理设计、失败恢复策略、和系统性地培养绕路能力。

同一个模型，同样的工具，只是多了一个"先分类错误再行动"的指令，执行效果天差地别。这不需要更强的模型，需要的是更好的工程。

当前的 AI Agent 生态有一个普遍的误区：把 Agent 当成更聪明的 Chatbot 来做。 给它更好的推理能力、更长的上下文窗口、更多的工具——这些当然有用，但它们解决的是"聪明程度"的问题，不是"靠谱程度"的问题。

一个靠谱的 Agent 和一个聪明的 Agent 的区别是什么？

聪明的 Agent 知道很多正确答案。靠谱的 Agent 在不知道答案时能快速找到另一条路。

后者需要的不是更大的模型，而是更好的错误恢复设计。这就是 Bypass 存在的意义。

八、如果你是 Agent 使用者

不管你用的是什么框架——Claude Code、OpenAI Codex、Cursor、OpenClaw、还是别的——这篇文章和这个 Skill 背后的原理对你都适用。

三个实用建议：

  1. 遇到 Agent 反复报错时，主动切断重试循环。 直接给出替代指令，比如"跳过这个工具，用 bash 手动执行 xxx"。不要反复说"再试一次"——这是在强化锚定。
  2. 提供结构化的错误判断线索。 你可以说"这个错误是硬限制，重试没用，换个方法"。帮模型跨过"可恢复 vs 不可恢复"的判断节点。
  3. 长对话中注意上下文锚定。 如果模型已经在错误的路径上走了好几轮，开新对话往往比在原对话中纠正更有效。新的上下文窗口没有锚定效应。

或者，更简单的方法——装个 Bypass，让 agent 自己学会判断。
