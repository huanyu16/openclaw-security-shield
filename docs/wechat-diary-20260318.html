AI 协作生图

AGENT · 一个有趣的 Agent-to-Agent 实战

发生了什么

下午想给 JOJO 角色画张图，于是让助手和 liblib.tv 的 AI 创作平台打了一交道。这一段经历比前面几步都更有意思，因为它让我第一次直观感受到了 Agent 之间如何协作

事情的起点很简单：我在 clawhub.ai 上找到了 LibTV 的 Skill 包，下载 zip 安装。安装完毕后，助手就多了一个能力——可以和 liblib.tv 的后端 AI Agent 对话

14:28 LibTV Skill 安装完毕，开始第一次实际调用

整个过程是这样的：

1️⃣ 我说了一句话："帮我画一张 JOJO SBR 风格的 Johnny 角色立绘"

2️⃣ 我的 AI 助手（前端 Agent）接到了这个需求，但并没有自己去理解"JOJO SBR 风格"是什么、该怎么构图、该用什么模型

3️⃣ 它只是把我的原话原封不动地打包，通过 API 发给了 liblib.tv 后端的另一个 AI Agent

4️⃣ 后端 Agent 收到消息后，自主拆解了任务：理解 JOJO 美学风格、选择合适的生成模型（Seedream 5.0）、编写精准的 prompt、设定构图参数

5️⃣ 生成完成后，前端 Agent 帮我把结果取回来、下载到本地、展示给我

我的助手不是在"画画"
它是在"替我和另一个AI传话"
前端理解意图 → 后端专业执行 → 前端取件交付
📦 LibTV Skill

Agent-to-Agent 的实际工作方式

LibTV 是 LiblibAI 推出的 AI 创作平台，专门为 Agent 设计了交互接口。后端 Agent 覆盖文生图、文生视频、图生视频、视频续写、局部编辑、风格迁移等全链路能力，掌握 Seedance 2.0、Kling 3.0、Midjourney、Seedream 5.0 等主流模型。安装 Skill 后，用户的 AI 助手通过 OpenAPI 创建会话、发送消息、轮询进度、下载结果，形成完整的 Agent 间协作闭环

后来我又试了几次，让助手同时调用 智谱 CogView-3 和 Liblib 两个平台画同一张图，对比两边的效果。两个 Agent 各自用自己擅长的方式工作，结果放在一起看，差异非常直观

Agent-to-Agent

INSIGHT · 当 AI 开始和 AI 协作

这到底是什么

今天的第 5 件事（AI 协作生图）让我第一次亲眼看到 Agent-to-Agent 是怎么工作的。事后想想，这件事比单纯的"装了个生图技能"有意义得多

简单说，Agent-to-Agent 就是：让 AI 助手去调用另一个 AI 服务。不是人去操作两个系统，而是你的 AI 助手替你和另一个 AI 打交道

具体怎么实现的？

分三层来看：

第一层：技能安装（Skill）
从 clawhub.ai 下载 LibTV 的 Skill 包，安装到本地。这个 Skill 本质上是一套API 调用脚本 + 行为规范文档（SKILL.md）。安装后，我的助手就知道"遇到生图需求时，该怎么跟 liblib.tv 通信"

第二层：会话创建与消息传递
我下达需求后，助手通过 Python 脚本调用 LibTV 的 OpenAPI，创建一个 AI 会话（类似微信聊天）。然后把我的原始需求原封不动作为消息发到这个会话里。这边是 HTTP 请求，那边是另一个 AI Agent 在接收

第三层：轮询与取件
消息发出去后，助手每隔几秒查询一次会话状态。后端 Agent 完成生图后，结果会出现在会话消息里。助手识别到结果 URL，自动下载到本地，展示给我

整个过程，我只需要说一句话。剩下的通信、等待、取件，全部由两个 Agent 之间自动完成

这种工作形式的本质意义是什么？

用一个不太精确但很好理解的类比：你有一个私人助理，你的助理认识一个画师

你不需要知道画师住在哪、用什么颜料、怎么沟通。你只需要跟你的助理说"帮我找他画张图"。你的助理知道该打哪个电话、怎么描述你的需求、什么时候去取画、取回来怎么交给你

Agent-to-Agent 的本质就是这件事的数字化版本。你拥有一个"认识所有人"的 AI 助理，每个专业领域都有一个对应的 AI 服务。你的助理负责理解你的意图、选择对的人、翻译需求、取回结果。你只需要跟你的助理说话

分权的价值：前端 Agent 不需要懂画画，后端 Agent 不需要懂你的上下文。各自做各自最擅长的事，通过标准化的 API 接口协作。这跟人类社会的分工逻辑完全一致

组合的威力：今天生图找 Liblib，明天做深度研究用另一个 Agent，后天写代码再换一个。你的私人助理可以接入无数个专业 Agent，每个都是各自领域的专家。能力的边界只取决于"你的助理认识谁"

真正的门槛：不在技术实现（API 调用很简单），在于每个后端 Agent 的专业能力有多强。LibTV 的后端 Agent 为什么能"一句话生短剧"？因为它背后有一整套对视频创作领域的理解：分镜拆解、镜头语言、模型选择、prompt 工程。这些是前端 Agent 做不好的，也不应该让前端去做

Agent-to-Agent 的本质
不是"更聪明的AI"
是"更合理的分工"
让每个 Agent 只做自己最擅长的事，通过接口协作完成单靠任何一个都做不好的复杂任务

INSIGHT · 今天的三条经验 + 一个新认知

一天下来做了7件事，前面6件是工具链搭建，最后1件是认知升级

搭工具链的过程很具体：生成内容、部署分享、配置系统、扩展技能、调通 API。每一步都是一个坑，踩完之后你就有了可以持续运转的个人 AI 工作站

但真正让我觉得有意思的是第 7 件事——Agent-to-Agent。当你的 AI 助手开始和另一个 AI 协作，你不再是在"用一个工具"，而是在经营一个协作网络。你的助手就是你的节点，每个后端 Agent 都是另一个节点，节点之间通过 API 连接。你的能力边界，取决于你的网络有多广

一个人 + 一个 AI 助手 + N 个专业 Agent
就是一个小型创意工作室
