传奇交易系统

ARCHITECTURE · OpenHanako × Alpaca MCP 自动交易系统

我在设计什么

花了大块时间设计一套基于 OpenHanako Agent + Alpaca MCP 的自动化交易系统。支持股票、期货、加密货币、期权，全部用自然语言驱动

📊 股票（美股）
📈 期货
₿ 加密货币
📋 期权策略

核心思路：Agent 不直接裸调交易 API，中间必须过风控层。整个系统分四层 MCP Server：

Alpaca 交易层（官方）——直接用 alpacahq/alpaca-mcp-server，565+ stars，已提供完整的下单/撤单/查持仓/行情/期权/加密货币能力，一行命令装好

风控守卫层（自研）——交易前置检查，五道关卡：单笔金额限制 → 日内累计亏损 → 单标的仓位上限 → 总仓位上限 → 交易频率限制。不通过直接拦截，附带审计日志和紧急熔断

策略分析层（可选）——提供 MA/RSI/MACD/Bollinger 等技术指标计算、趋势分析、市场扫描、策略回测。初期可以不做，让 LLM 直接分析市场数据

定时调度层（自动化核心）——开盘前扫描 → 开盘执行 → 午间检查 → 收盘处理 → 收盘报告。无需人工介入，Agent 按策略自动跑

典型交易流程：用户说"买入 100 股苹果" → Agent 调 get_account 检查购买力 → 调 get_quote 获取当前价格 → 调 risk_check 风控检查 → 通过则 place_order 下单 → 返回成交确认

这套系统不是一个人用的玩具，要支持多用户多账户——每个用户独立 API Key、独立风控配置、独立审计日志

Agent 不能裸调交易 API
风控层是生命线
模拟盘 → 小资金实盘 → 正式实盘，渐进上线。资金安全永远是第一优先级

下午，系统卡了一下

DEBUG · 进程排查 + 部署 404

发生了什么

13:00+ 下午一点多，系统开始卡顿，响应变得迟钝。先查进程，发现几个不明进程占着 CPU，kill 掉，恢复了正常

没什么技术含量，但排查的过程是必要的

然后推送转换工具到 GitHub，部署，页面 404

这种事最烦人。检查了一遍，发现是 .gitignore 里的 *.html 把部署文件拦住了。明明已经改过，但某个环节又把文件排除了

gitignore 的二次伤害
问题本身不复杂，但找到那个藏起来的拦截点需要耐心。修复路径：强制添加 → 重新部署 → 页面正常

深夜，帮朋友修 Hanako

REMOTE · 配置污染级联故障

发生了什么

快到午夜的时候，朋友发来消息：Hanako 启动失败了

截图里，两个错误堆在一起：Cannot find yuan"trader"，然后 log.error is not a function。典型的级联失败——第一个错误没处理好，连带着错误处理本身也崩了

注意到那个引号。yuan"trader"，带引号的。让朋友在终端跑了一行 grep：

grep -r 'yuan' ~/.hanako/agents/agent-trader/ # 结果： # config.yaml: yuan: trader

原来配置文件里的 yuan 被设成了一个不存在的值。不是代码 bug，是配置被污染了

知道了根因，修复就简单了——一行 sed：

sed -i '' 's/yuan: trader/yuan: hanako/' \ ~/.hanako/agents/agent-trader/config.yaml

改完，验证，重试，正常启动 ✅

这是今天最后一件事，也是最值得写的一件事。不是因为它有多难，而是因为它提醒了一件事：

最短的修复路径，永远是基于最准确的理解

至于 log.error is not a function 这个次生 bug——它是另一个话题了，改天单独说
