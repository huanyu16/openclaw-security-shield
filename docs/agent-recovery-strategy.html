Agent · 深度分析

同样的 Hanako，同样的工具，为什么执行能力天差地别？

一个安装技能的需求，一道 25 stars 的安全拦截，两种截然不同的推理路径。拆开来看，问题不在模型智商，在别的地方。

2026.03.19 · 约 8 分钟阅读

一、发生了什么

两个用户，各自跑着 Hanako，都接到同一个任务：安装 mindverse/Second-Me-Skills 到 Claude Code。

安装过程使用了 install_skill 工具，这个工具内置了安全策略——目标仓库必须达到 25 stars 才允许安装。目标仓库只有 17 stars。

❌ Hanako 实例 A

  • 首次尝试被拒（star 不够）
  • 用户再次要求，同参数重试
  • 依然被拒，继续重试
  • 反复撞墙后陷入死循环
  • 用户质疑后，才最终绕路用 git clone

✅ Hanako 实例 B

  • 首次尝试被拒（star 不够）
  • 立即判断这是硬限制，重试无意义
  • 跳过 install_skill，直接走 git clone
  • 克隆到 skills 目录，创建符号链接
  • 一步到位，无多余尝试

注意关键前提：两个实例用的是同一个工具、同一套约束、同一个安全门槛。 变量是完全相同的。唯一的区别是模型在收到"被拒绝"信号后的推理路径。

二、根本原因：错误恢复策略的差异

这不是"谁更聪明"的问题。模型智商在同一代产品中差异很小。真正的差异在于面对失败信号时的推理策略——学术上叫 Error Recovery Strategy。

2.1 两种推理链的对比

实例 A
install_skill → 被拒
↓ "再试一次也许能行"
install_skill → 被拒
↓ "用户坚持了，再试"
install_skill → 被拒
↓ "用户质疑了，才想起来换路"
git clone → 成功
实例 B
install_skill → 被拒
↓ "硬限制，重试无效，换路"
git clone → 成功

核心分歧就在一个判断节点上：

收到"操作被拒绝"信号后，这个失败是可恢复的（换个参数再试）还是不可恢复的（硬限制，必须换方法）？

实例 A 把它当成可恢复的，所以反复重试。实例 B 把它当成不可恢复的，所以立刻绕路。

2.2 为什么会出现这种分歧？

三个因素叠加：

第一，System Prompt 的隐性引导。 虽然都是 Hanako 的 prompt，但不同版本的措辞会微妙地影响模型的行为倾向。有些版本的 prompt 偏向"遵循工具规范"，有些偏向"以结果为导向解决问题"。一个写着"工具返回错误时请向用户解释原因"的 prompt，比"遇到障碍时立即寻找替代方案"的 prompt，更容易让模型陷入重试循环。

第二，温度参数（Temperature）。 推理路径的选择有一定随机性。温度较高时模型更容易跳出"常规路径"去尝试替代方案，温度较低时倾向于选择"最安全、最符合上下文"的路径——而"再试一次"往往被模型认为是"最安全"的响应。

第三，上下文窗口的内容密度。 如果对话历史中积累了大量的重试记录，模型会被锚定在"当前正在使用 install_skill"这个上下文里，更难跳出。这叫 context anchoring（上下文锚定），是 LLM 推理中一个被广泛观察到的现象。

三、这不是 Hanako 的问题，是整个 Agent 领域的共性问题

这个案例暴露的是当前 AI Agent 设计中的一个结构性缺陷：工具报错和工具硬限制没有区分。

当工具返回错误时，模型收到的是一个统一的"失败"信号。但它无法判断：

  • 这个失败是因为临时问题（网络超时、速率限制）→ 应该重试
  • 这个失败是因为参数错误（格式不对、缺少字段）→ 应该修正参数
  • 这个失败是因为硬策略拦截（安全门槛、权限不足）→ 必须换路

理想的设计是工具在返回错误时，同时返回一个错误类型标记（比如 retryable / fixable / blocked），让模型能据此做出正确的恢复策略。但目前大多数 Agent 框架都没有做这个区分。

一个更好的设计应该是这样的：

install_skill 返回：
{
  "status": "blocked",
  "reason": "仓库 star 数(17) 低于安全门槛(25)",
  "is_retryable": false,
  "suggestion": "此为硬性安全策略，重试不会改变结果。可考虑使用 git clone 手动安装。"
}

有了这个结构化信息，即使是实例 A 也不会再去重试了。问题不在模型，在工具的报错设计。

四、给 Agent 使用者的实用建议

理解了这个机制，你可以更有效地使用任何 AI Agent：

1. 遇到 Agent 反复报错时，主动切断重试循环。 不要反复说"你试过了吗？""再试一次"。直接给出替代指令，比如"跳过这个工具，用 bash 手动执行 xxx"。

2. 提供结构化的错误判断线索。 你可以说"这个错误是硬限制，重试没用，换个方法"。帮模型跨过"可恢复 vs 不可恢复"的判断节点。

3. 长对话中注意上下文锚定。 如果模型已经在错误的路径上走了好几轮，开启新对话往往比在原对话中纠正更有效。新的上下文窗口没有锚定效应。

五、更根本的问题：如何让智能体变得更"能干活"？

上面的案例和分析都指向一个更大的命题：我们到底怎么让 AI Agent 从"能用"变成"好用"，从"偶尔灵光"变成"稳定靠谱"？执行力不是玄学，它是可以被系统化设计的。

5.1 第一层：工具层的报错要"说人话"

前文已经提到，工具返回错误时不区分类型，是导致模型反复撞墙的直接原因。但问题比这更深：大多数工具的错误信息本身就不适合模型消费。

人类开发者看报错，靠的是经验——看到 403 立刻想到权限，看到 timeout 立刻想到网络。模型没有这个直觉。它需要工具把错误拆成结构化的、无歧义的信号：

{
  "error_type": "hard_block",      // retryable | fixable | hard_block
  "reason": "安全策略限制：仓库 star 数低于门槛",
  "user_actionable": true,
  "alternative": "git clone + 手动配置"
}

这不仅仅是"加个字段"。这是在设计 Agent 的错误通信协议。工具应该把模型当成一个需要精确指令的协作者，而不是一个能猜透你心思的搭档。

5.2 第二层：Prompt 里要写"失败策略"，不只是"成功路径"

现在的 System Prompt 绝大多数只描述"你应该怎么做"，极少描述"你做失败了应该怎么恢复"。这就像给员工写操作手册，只写了正常流程，没写应急预案。

一个好的 Agent Prompt 应该包含：

  • 工具选择优先级——A 失败时优先尝试 B，B 失败再尝试 C
  • 失败类型判断标准——哪些错误值得重试，哪些必须换路
  • 重试上限——同一个工具同一个参数，最多试几次
  • 降级策略——高级工具不可用时，用基础工具的什么组合来替代

这不是在教模型"变聪明"，是在给模型一个决策框架。有了框架，它不需要每次都从零推理"我应该怎么办"，而是直接套用经过验证的策略模式。

5.3 第三层：给模型"跳出当前路径"的机制

上下文锚定是 Agent 执行力最大的隐形杀手。模型一旦在某个路径上走了几步，它就很难跳出来——即使这条路已经被证明走不通。

解法有几个方向：

a) 内置反思步骤。 在 prompt 中要求模型每隔 N 步做一次"路径检查"：当前方法是否有效？有没有被卡住？是否需要换方向？这类似于人类工作中定期问自己"我在做正确的事吗"。

b) 外部触发器。 让系统在检测到模型连续失败时，自动注入一段"重置上下文"的指令。比如连续三次报错后，系统自动追加："你已经用 install_skill 尝试了 3 次均失败。请停止使用此工具，改用替代方案。" 这相当于给模型一个"被踹一脚"的外部刺激。

c) 工具使用审计。 记录每次工具调用的参数和结果，在模型下一步决策前把它作为上下文注入。模型看到"同一个工具、同一组参数、三次同样结果"的审计记录，比单纯看到最后一次报错更容易触发"换路"的判断。

5.4 第四层：把"绕路能力"作为核心能力来培养

最根本的一点：Agent 的价值不在于它能顺利执行理想路径，而在于理想路径被堵死时它能不能找到另一条路。

这需要三件事：

  • 工具箱足够丰富。 模型至少需要具备基础的 bash、文件读写、网络请求能力，才能在任何工具失败时兜底。
  • 工具之间可以互相替代。 安装技能可以用 install_skill，也可以用 git clone + 手动配置。发搜索请求可以用 web_search，也可以用 bash + curl。没有哪个功能只能通过唯一工具实现。
  • prompt 明确鼓励"绕路思维"。 "如果工具 A 不可用，永远有工具 B 可以替代"——这种信念要写进 system prompt，成为模型的底层行为模式，而不是靠它自己"碰巧想到"。

5.5 一句话概括

让 Agent 更能干活，关键不是更大的模型、更多的工具，而是更好的错误处理设计、更明确的失败恢复策略、以及系统性地培养"绕路能力"。执行力的本质是鲁棒性——在最坏条件下依然能交付结果的能力。

六、总结

同样的工具，同样的约束，不同的推理路径。差异的本质不是模型智商，而是模型在面对失败信号时，能否快速区分"可恢复错误"和"不可恢复错误"，并据此选择正确的恢复策略。这是当前 AI Agent 领域一个尚未被充分重视的结构性问题。

而作为使用者，理解这一点，能让你少很多"你怎么这么笨"的烦躁，多一种"我知道怎么帮你跳过这个坑"的从容。
